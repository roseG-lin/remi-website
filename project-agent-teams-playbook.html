项目概述

不是一个执行者，而是一个智能编排决策中心

Agent Teams Playbook 是一个专为 Claude Code 设计的智能编排系统，作为决策中心而非执行者， 它负责决策用哪个工具、怎么组合、何时切换策略，而执行交给专业工具和 subagent。 通过五件套工具链实现复杂任务的自动化编排，让任务变得可追踪、可管理、可交付。

核心理念

Agent Teams = "并行处理 + 结果汇总"

并行处理

多个 Agent 同时工作，每个 Agent 独立处理不同任务模块，大幅提高执行效率

上下文独立

每个 Agent 拥有独立的思考空间和上下文窗口，避免信息干扰和混淆

结果汇总

统一收集和整合各 Agent 的输出，压缩返回主会话，形成完整交付物

工作流程

6 阶段自动编排流程

0

规划准备

调用 planning-with-files 创建三个核心文件

  • task_plan.md - 任务计划和阶段追踪
  • findings.md - 研究发现和知识积累
  • progress.md - 执行日志和进度记录
1

任务分析 + Skill 发现

先质疑再执行，输出任务总览，执行 Skill 完整回退链

  • 本地 Skill 扫描 → 外部 Skill 搜索 → 通用 Subagent
  • 输出任务目标、预期结果、验收标准
2

团队组建

输出团队蓝图，分配角色、职责、模型

  • 架构师、实现者、审查员等角色分配
  • opus/sonnet/haiku 模型选择
3

并行执行

Skill 任务用 Skill 工具调用，通用任务用 Task 工具生成 subagent

  • 每个 agent/skill 完成后汇报结果
  • 进度实时写入 progress.md
4

质量把关

对照验收标准逐项检查，产品打磨

  • 边界处理、专业度、完整性检查
  • 测试覆盖率检查（TDD 任务要求 ≥ 80%）
5

结果交付

输出执行报告，提供部署移交文档

  • 总任务数、各 Agent 结果、汇总结论
  • 最终状态更新到 task_plan.md
编排场景

5 大编排场景

1

提示增强

适用条件：简单任务，1-2 步

核心策略：优化单 agent 提示词，不拆分不组队

2

Skill 直接复用

适用条件：单个 Skill 可完全解决

核心策略：直接调用匹配的 Skill

3

计划 + 评审

适用条件：中等/复杂任务（默认场景）

核心策略：出计划 → 确认 → 并行执行 → 验收

4

Lead-Member

适用条件：需要明确团队分工

核心策略：Leader 协调，Member 并行执行

5

复合编排

适用条件：复杂任务，无固定模式

核心策略：动态组合上述场景，按阶段切换策略

协作模式

两种协作模式

模式 通信方式 适用场景 启动方式
Subagent 子 agent → 主协调器单向汇报 并行独立任务 Task 工具
Agent Team 成员间可双向通信 需要协作的复杂任务 TeamCreate + Task(team_name)

选择原则：任务间无依赖用 Subagent（简单高效），任务间需要协调用 Agent Team（功能更强但成本更高）

执行底线

9 条设计原则

每一条背后都有一个翻车故事，确保系统稳定可靠运行

0

强制使用 planning-with-files：任何复杂任务必须先创建计划文件

1

强制执行 Skill 完整回退链：本地扫描 → find-skills → 通用 subagent

2

先目标，后组织结构：任务不清晰时先澄清

3

队伍规模由任务复杂度决定：并行 Agent 建议不超过 5 个

4

关键里程碑必须有质量闸门：不能 Agent 干完就算完

5

不默认任何外部工具可用：执行前先验证

6

成本只是约束，不是承诺：不做不切实际的预估

7

遇到问题给选项：提供重试/跳过/终止选项

8

危险操作必须确认：删除、重构等必须用户点头

实战示例

自动执行流程

用户输入

"帮我重构认证模块，拉团队搞"

阶段 0

调用 planning-with-files → 生成 task_plan.md + findings.md + progress.md

阶段 1

任务分析 + Skill 发现 → 重构认证模块，预计 4 个 Agent → 找到 brainstorming, test-driven-development → 选定场景：场景 3

阶段 2

组建团队 → 架构师 (opus) + 实现者 A (sonnet) + 实现者 B (sonnet) + 审查员 (sonnet)

阶段 3

并行执行 → 架构师设计方案 + 实现者 A TDD 实现 + 实现者 B TDD 实现 + 审查员 Code Review

阶段 4

质量把关 → 验收标准检查 + 测试覆盖率检查 (≥ 80%)

阶段 5

结果交付 → 执行报告 + 最终状态更新到 task_plan.md
